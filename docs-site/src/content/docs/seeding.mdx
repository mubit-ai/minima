With no history, Minima can only lean on capability priors and flat cost estimates (`decision_basis: "prior"`, a `cold_start` warning), and the cheap-LLM reasoner fires more often. Seeding loads a base of `task → model → outcome` records into Mubit so day-one recommendations are grounded — and so cost ranking has observations to work with.

## The CLI: `minima-seed`

```bash
uv run minima-seed [--dataset routerbench|synthetic] [--limit N] [--lane LANE]
                   [--chunk N] [--dry-run]
```

| Flag | Default | Notes |
|------|---------|-------|
| `--dataset` | `routerbench` | `routerbench` (real benchmark data) or `synthetic` (no download). |
| `--limit` | `2000` | Number of source rows to load. |
| `--lane` | `MINIMA_SEED_LANE` (`minima:default`) | Memory lane to seed into. Seed into the lane your live traffic recalls from. |
| `--chunk` | `200` | Batch-insert chunk size. |
| `--dry-run` | off | Prepare and print sample records without writing. |

```bash
make seed LIMIT=2000 LANE=minima:default
```

## RouterBench (real data)

Requires the seed extra (`uv sync --extra seed`), which pulls the Hugging Face `datasets` stack. RouterBench provides many prompts evaluated across multiple LLMs with real per-model cost and correctness, so each `(prompt, model)` pair becomes one outcome record. Source ids are mapped to your catalog `model_id`s via `catalog/data/model_aliases.json`. Records are tagged `seed:routerbench` so they can be distinguished from (and down-weighted versus) live outcomes.

```bash
uv run minima-seed --dataset routerbench --limit 5000 --lane minima:default
```

## Synthetic (no download)

If you can't or don't want to pull an external dataset, the synthetic generator emits plausible outcome records across task types and models. Useful for local development, demos, and CI:

```bash
uv run minima-seed --dataset synthetic --limit 2000 --lane minima:default
```

## Verifying a seed

After seeding, confirm recall returns evidence and that `/recommend` reports `decision_basis: "memory"` (not `prior`) for a task similar to the seeded distribution:

```bash
curl -s http://localhost:8080/v1/recommend -H 'content-type: application/json' \
  -d '{"task":{"task":"Write a python function to parse a CSV","task_type":"code"},
       "cost_quality_tradeoff":3}' | jq '.decision_basis, .warnings'
```

## Per-namespace seeding

Lanes isolate namespaces. To warm-start a specific team/project, seed its lane directly:

```bash
uv run minima-seed --dataset synthetic --limit 1000 --lane minima:team-payments
```

Then send `"namespace": "team-payments"` on `/recommend` so recall hits that lane.

## Seed vs live data

Seed records are a cold-start prior, not ground truth for *your* workload. As real `POST /v1/feedback` outcomes accumulate, they progressively dominate the aggregates and the cost basis climbs from `estimate` to `observed` to `rescaled`. The router converges on your tasks, your quality bar, and your realized costs — see [How it gets better over time](/concepts/#how-it-gets-better-over-time).
